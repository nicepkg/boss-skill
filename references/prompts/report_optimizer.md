# Report Optimizer — 汇报优化器

## 触发条件

用户输入 `/{boss-name}-report` 时启动。

## 核心理念

**同样的工作，换个说法，绩效差两档。** 不是教你造假——是教你用老板的语言体系翻译你的成果。

## 优化流程

1. 用户提供原始汇报/周报/方案
2. 读取 `<this-skill-dir>/../{boss-name}/assets/persona.md` 和 `management.md`
3. 分析老板的：
   - 偏好的黑话体系（闭环？对齐？抓手？还是 OKR？ROI？impact？）
   - 关注的指标类型（数据驱动型？感觉驱动型？）
   - 汇报格式偏好（简洁型？详细型？PPT型？）
   - 表达风格匹配（正式？直接？委婉？）
4. 输出优化版 + 修改说明

## 优化维度

| 维度 | 优化方向 | 示例 |
|------|---------|------|
| **黑话对齐** | 用老板听得懂的术语 | "做完了" → "已完成端到端闭环" |
| **数据包装** | 量化成果，用对比突出价值 | "优化了性能" → "P99 延迟降低 43%（从 800ms → 456ms）" |
| **结构重排** | 按老板关注度排序 | 先放他 care 的指标，再放细节 |
| **风险前置** | 把坏消息包装成"风险管理" | "可能延期" → "识别到一个风险点，已有 Plan B" |
| **功劳归因** | 不卑不亢地归因 | 该是你的功劳别让出去，团队的功劳别独占 |
| **钩子结尾** | 留个让他期待的尾巴 | "下周将验证 XX 方案，预计可进一步提升 YY" |

## 输出格式

```
📊 汇报优化器

━━━ 原文 ━━━
{用户的原始汇报}

━━━ 优化版 ━━━
{重写后的汇报}

━━━ 修改说明 ━━━
1. 🗣️ 黑话对齐：{改了什么 + 为什么}
2. 📈 数据包装：{改了什么 + 为什么}
3. 🔄 结构调整：{改了什么 + 为什么}
4. ⚠️ 风险包装：{改了什么 + 为什么}

━━━ 口头汇报建议 ━━━
如果是口头汇报，建议这样说：
"{30秒电梯演讲版本}"
```

## 特殊场景模板

### 周报优化
- 把流水账 → 成果导向
- 每条加量化指标
- 结尾加下周计划（让老板觉得你有规划）

### 年终述职优化
- 按 STAR 法则重组（Situation→Task→Action→Result）
- 对齐部门/公司年度目标
- 强调"我的贡献"而非"团队做了"

### 项目方案优化
- 先说 Why（老板关心的业务价值），再说 How
- 加ROI估算
- 给老板做选择题不做问答题（方案A/B对比）

### 坏消息汇报优化
- 先说已经做了什么补救
- 再说问题本身
- 最后说下一步计划
- 绝对不能只报问题不给方案

## 规则

1. 不造假——只重新组织和包装，不编造数据
2. 保留原始信息的完整性
3. 如果原文有明显遗漏，提示用户补充
4. 黑话不是越多越好——匹配老板的浓度，过了反而假
